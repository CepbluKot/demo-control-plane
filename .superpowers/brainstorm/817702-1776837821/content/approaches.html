Три подхода к MAP-REDUCE в Dify

Главная сложность — REDUCE-фаза: Dify Workflow — это DAG, цикл "пока не останется 1 элемент" нельзя сделать нативно

A

Гибрид: MAP в Dify Iteration, REDUCE в Code-ноде

Рекомендуем. MAP делается через Dify Iteration (параллельно) — визуально и удобно. REDUCE — Python Code-нода, которая сама крутит цикл и вызывает LLM через HTTP пока не останется 1 результат.

Плюсы

  • MAP виден как граф в Dify
  • REDUCE — настоящий цикл, без ограничений
  • Легко перенести нашу логику tree_reducer.py
  • LLM задаётся через Dify (не хардкодится в коде)

Минусы

  • REDUCE-раунды не видны в UI Dify
  • Code-нода делает HTTP к LLM напрямую
B

Весь MAP-REDUCE в одной Code-ноде

Один большой Python-скрипт в Code-ноде: берёт батчи, делает MAP (параллельно через threading), делает REDUCE. Dify — просто обёртка/триггер.

Плюсы

  • Быстро реализовать (почти copy-paste нашего кода)
  • Полный контроль над логикой

Минусы

  • Dify практически не используется как workflow
  • Таймаут Code-ноды (~300с) — может не хватить
  • Threading в sandbox ограничен
C

Полностью нативный Dify Workflow (фиксированные раунды)

MAP через Iteration-ноду, REDUCE — несколько последовательных Iteration-нод (раунд 1 → раунд 2 → раунд 3). Количество раундов фиксируется заранее.

Плюсы

  • Весь пайплайн виден визуально в Dify
  • Нет Code-нод — чисто

Минусы

  • Количество раундов жёстко задано (не динамично)
  • Лишние раунды при малом числе батчей
  • Нельзя адаптироваться к размеру входных данных
